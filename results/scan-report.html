🔧 Remediation Recommendations

🚨

CRITICAL — Restrict User Object Access for Guest Profile

The User object is readable via both Aura and GraphQL APIs without authentication. In Salesforce Setup → Profiles → Guest User Profile → Object Settings → set User to No Access. Also review field-level security to ensure email, phone, and name fields are not exposed.

📄

HIGH — Lock Down ContentDocument & ContentVersion Guest Access

Files stored in Salesforce Files (Content) are accessible via guest session. Set ContentDocument and ContentVersion sharing to internal only. Review all ContentDocumentLink records shared with the Community. Use WITH SECURITY_ENFORCED in SOQL and apply Apex sharing rules.

⚡

HIGH — Disable or Restrict GraphQL uiapi for Guest Users

The /services/data/vXX.0/ui-api/records GraphQL interface exposes 19 objects beyond what Aura exposes. In Experience Cloud → Administration → Security → disable Lightning Data Service for guest. Alternatively restrict object permissions on the Guest User Profile for all objects in the GraphQL results.

🔐

HIGH — Restrict EntityDefinition and Profile Metadata Exposure

Exposing 286 EntityDefinition and 120 Profile records enables full org schema fingerprinting. Remove EntityDefinition read access from the Guest User Profile. This is used by attackers to craft precise SOQL injection payloads and map privilege levels.

📅

MEDIUM — Audit Calendar Record Sharing (2,000 records exposed)

2,000 Calendar records are accessible to guest users. Verify that no internal meetings, participant names, or sensitive scheduling data are included. If Calendar is not a required feature for the guest experience, remove sharing and object access from the Guest User Profile entirely.

💬

MEDIUM — Review LiveChatButton Configuration Exposure

440 LiveChatButton configuration records are exposed via GraphQL. These may reveal internal routing queues, skill names, and service channel IDs. Restrict LiveChatButton read access on the Guest User Profile or apply record-level sharing rules.

🧼

MEDIUM — Disable SOAP API for Guest/External Contexts

The SOAP API (/services/Soap/) is enabled and could be leveraged for credential-stuffing, metadata probing, or brute-forcing with username/password authentication. Disable SOAP API in the org's API settings if not required, or block via network policy / Connected App restrictions.

🔗

MEDIUM — Audit CSP Wildcard Trusted Origins

6 wildcard CSP entries (*.adobe.io, *.appdynamics.com, etc.) broaden the attack surface for supply-chain attacks. Replace wildcards with explicit subdomain entries. Audit all third-party scripts (GTM, BazaarVoice, AppDynamics) for subresource integrity (SRI) enforcement.

🔍

LOW — Verify Topic Record List URL

Manually open /producttesterprogram/s/recordlist/Topic/Default in an incognito browser to verify if a browseable table of records is rendered for unauthenticated users. If visible, add a guest-user-specific list view permission restriction or remove the public page entirely.

🔒

LOW — Apply Principle of Least Privilege to Guest User Profile

Run a full Guest User Profile audit in Salesforce Setup → Profiles → Site Guest User. Remove read access for all objects not explicitly required for the guest experience (product browsing, registration form). Enable Enhanced Object Permissions enforcement for Experience Cloud sites.

🔐

HIGH — Add HTTP Strict-Transport-Security (HSTS) Header

The site does not send an HSTS header, allowing it to be loaded over HTTP. Add Strict-Transport-Security: max-age=31536000; includeSubDomains via the CDN/WAF in front of the Salesforce domain, or via a Custom Header in Experience Cloud Security Settings.

🖼️

MEDIUM — Add X-Frame-Options and Referrer-Policy Headers

Missing X-Frame-Options: SAMEORIGIN allows clickjacking on login/form pages. Missing Referrer-Policy leaks full URLs to analytics providers. Add both in Setup → Security → HTTP Response Headers or via CDN rules.

🍪

MEDIUM — Enforce Secure Cookie Flags

Session cookies missing HttpOnly or SameSite=Strict expose users to XSS token theft and CSRF. In Salesforce Setup → Session Settings, enable Lock sessions to the IP address from which they originated and ensure all custom cookies set by LWC/Apex are created with HttpOnly; Secure; SameSite=Strict.

⚡

MEDIUM — Restrict Lightning Screen Flow Guest Access

FlowController/getFlowMetadata is accessible to unauthenticated users. Audit all Screen Flows exposed on guest pages for DML operations running in system context. Set Flow Run Mode to Default (User Context) and add a Guest user check at the first decision element.

↪️

MEDIUM — Fix Open Redirect via login retURL

The /login?retURL= parameter accepts arbitrary external URLs. Validate retURL against an allowlist of your own domain(s) before redirecting. Reject any value that does not start with your Salesforce site origin to prevent phishing via seemingly-legitimate links.
